# Test Vectors

This document describes the official PostQuantum.Sqlite test vector
corpus living under
[`tests/PostQuantum.Sqlite.Tests/Vectors/`](../tests/PostQuantum.Sqlite.Tests/Vectors).

The corpus exists so an **independent** `.pqsm` implementation can
prove conformance to the spec without trusting this code. If your
implementation passes every assertion below, it accepts the same
manifests and rejects the same malformed inputs we do.

## Layout

```
Vectors/
├── manifest.json                                  ← index, describes each vector
├── positive/
│   └── kem-single-recipient/
│       ├── input.pqsm                             ← the manifest under test
│       ├── recipient.encap.key                    ← 1184 bytes (ML-KEM-768)
│       ├── recipient.decap.key                    ← ~2400 bytes
│       ├── signer.public.key                      ← 1952 bytes (ML-DSA-65)
│       ├── database.salt                          ← 16 bytes
│       └── expected.dek                           ← 32 bytes
└── negative/
    ├── duplicate-recipient-field.pqsm
    ├── noncanonical-order.pqsm
    ├── trailing-bytes.pqsm
    ├── truncated-nonce.pqsm
    ├── unknown-toplevel-field.pqsm
    └── wrong-salt-length.pqsm
```

`manifest.json` is the machine-readable index:

```json
{
  "version": 1,
  "positive": [ { "name": "...", "path": "...",
                  "recipientEncapsulationKey": "...",
                  "recipientDecapsulationKey": "...",
                  "signerPublicKey": "...",
                  "databaseSalt": "...",
                  "expectedDek": "..." } ],
  "negative": [ { "name": "...", "path": "...",
                  "description": "...",
                  "expectedErrorContains": "..." } ]
}
```

`expectedErrorContains` is a case-insensitive substring of the error
your implementation should produce. The exact wording is implementation-
defined; what is **normative** is that an error is produced at all.

## Positive: `kem-single-recipient`

A minimal well-formed manifest. One ML-KEM-768 recipient, ML-DSA-65
signed, revision 1.

To accept this vector your implementation MUST be able to:

1. Strict-parse the canonical CBOR (`docs/SPEC.md` §2–§4).
2. Verify the ML-DSA-65 signature using the supplied
   `signer.public.key` over the canonical CBOR of fields 1–7 of the
   manifest (the signed payload).
3. Compare the manifest's `database-salt` field against the supplied
   `database.salt` and confirm byte-equality.
4. Find the recipient entry whose `fingerprint` equals the first 16
   bytes of `SHA-256(recipient.encap.key)`.
5. Decapsulate the recipient's `kem-ciphertext` using
   `recipient.decap.key` (ML-KEM-768).
6. Derive a 32-byte KEK via HKDF-SHA256 with:
   - `ikm`  = decapsulated shared secret
   - `salt` = `database.salt`
   - `info` = `"PostQuantum.Sqlite/kek" || 0x00 || version (int32 BE) || 0x00 || "ML-KEM-768" || 0x00 || fingerprint`
   - `L`    = 32
7. AES-256-GCM decrypt the recipient's `wrapped-dek` (32-byte ciphertext
   ‖ 16-byte tag) using the KEK, the recipient's `nonce`, and AAD =
   `database.salt || fingerprint`.
8. Confirm the resulting 32-byte plaintext equals `expected.dek`.

If any step fails, the implementation is non-conformant.

## Negative vectors

Each MUST be rejected. The current corpus exercises one
parser-rejection rule per file:

| Vector | Rule from `docs/SPEC.md` |
|---|---|
| `unknown-toplevel-field` | §2 — unknown fields hard-rejected (whitelist parser, no extensions in v1). |
| `duplicate-recipient-field` | §2 — canonical CBOR requires unique map keys. |
| `noncanonical-order` | §2 — canonical CBOR requires sorted map keys. |
| `trailing-bytes` | §2 — trailing bytes after the CBOR document are rejected. |
| `truncated-nonce` | §4 — AES-GCM nonce length is exactly 12 bytes. |
| `wrong-salt-length` | §3 — `database-salt` length is exactly 16 bytes. |

A conformant parser MUST surface a distinguishable error for each
case. The exact rejection sequence (e.g. whether duplicate keys are
caught by the CBOR layer or the application layer) is
implementation-defined, but the input MUST NOT silently parse.

## Regenerating the corpus

The committed bytes are the source of truth. Regeneration is for
maintainers who change the format and need to refresh the vectors:

```bash
# 1. In tests/PostQuantum.Sqlite.Tests/VectorGenerator.cs, change
#       [Fact(Skip = "Vector generator. ...")]
#    to
#       [Fact]
#
# 2. Run:
dotnet test tests/PostQuantum.Sqlite.Tests \
    --filter "FullyQualifiedName~VectorGenerator" -c Release
#
# 3. Put the Skip back. Inspect `git diff Vectors/` and commit if intended.
```

ML-KEM-768 keypair generation and ML-DSA-65 signing are randomized
under the .NET 10 BCL, so the regenerated bytes will not be
byte-equal to the committed ones — but they will be **semantically**
equivalent and will pass the same conformance assertions.

## How `VectorTests` exercises the corpus

[`tests/PostQuantum.Sqlite.Tests/VectorTests.cs`](../tests/PostQuantum.Sqlite.Tests/VectorTests.cs)
is one possible conformance runner. xUnit `[Theory]` cases iterate
through `manifest.json`; the positive case runs the eight steps
above end-to-end; the negative cases assert that
`PqSqliteManifest.Deserialize` throws a `PqSqliteException` whose
message contains the documented substring.

Port the assertions to your test framework of choice; the corpus
itself is language- and runtime-agnostic.
